1. 서버 구동 및 정지

매니저 포트설정

RDServer는 서블릿 버전과 데몬 버전으로 나누어지며 기본 서비스 포트와 서버 관리자와 통신할 수 있는 포트 2개를 필요로 합니다. 매니저 포트는 기본값으로 8089를 사용하지만 다른 어플리케이션에서 이미 사용 중이라면 다음 파일에서 변경할 수 있습니다.

n  데몬 버전

Rdserver/bin 밑에 있는 startup.bat(Windows 계열)이나 startup.sh(Unix/Linux 계열) 파일을 열어 RDSERVER_MANAGER_PORT의 포트 번호를 변경해 줍니다.

RDSERVER_MANAGER_PORT=8089

n  서블릿 버전

                                                RDServer/rdagent.jsp파일을 열어 다음과 같은 라인의 값을 변경합니다.

                                                private static int managerport = 8089;

 

서블릿 서버 구동

서블릿 버전은 WAS 구동 후 서버 설치 URL을 다음과 같이 호출하면 자동으로 구동됩니다. 기본 포트는 WAS에서 설정한 서비스 포트를 사용합니다.

http://serverip:port/RDServer/rdagent.jsp

1) Viewer 또는 Editor 를 이용한 서버 데이터베이스 접속 시

2) 웹 브라우저에서 http://serverip:port/RDServer/rdagent.jsp 호출 시

참고) 서블릿 서버를 구동한 상태에서 RDServer/bin/startup.sh 또는 startup.bat 를 실행하면 BindException이 발생할 수 있으므로 서블릿 버전 운영 시에는 실행하지 않습니다.

데몬 서버 구동

인스톨러를 이용하여 설치를 마치고 곧바로 자동 구동이 가능하나 수동으로 시작할 때는 RDServer/bin/startup.bat 또는 startup.sh를 실행합니다. 서비스 기본포트는 8282입니다.

(UNIX, Linux계열: RDServer/bin/startup.sh를 chmod +x하여 실행 모드로 수정 후 실행)

그림 (4) 정상적으로 서버를 종료시킬 때의 예

 

유닉스/리눅스 계열: RDServer/bin/shutdown.sh를 chmod +x하여 실행 모드로 수정 후 실행합니다.

> shutdown.sh

위와 같이 shutdown.sh을 실행시키면 구동 중인 서버가 종료됩니다. 정지 명령을 내릴때는 처음 서버를 실행한 사용자나 그와 동일한 권한이 있는 사용자가 shutdown 시킬 수 있습니다. 그렇지 않을경우 permission denied 오류가 발생할수 있습니다.

그림 (5) 권한이 없는 사용자가 shutdown.sh 를 실행시킨 경우

startup.sh 를 실행시킬 때 RDServer/demon.pid에(파일이 없으면 자동생성) 서버에 대한 프로세스 아이디를 저장하게 됩니다. 종료되지 않을 경우는 이 파일에서 pid를 확인하고, 그림 (7) 처럼 kill 명령으로 종료시킵니다.

관리자 로그인

RDServer를 관리하기 위해 사용자 인증과정을 거쳐야 합니다. 왼쪽 서버 트리에 등록된 RDServer를 더블 클릭하면 로그인 할 수 있도록 아래 그림(1)과 같이 로그인창이 표시됩니다. 처음 로그인한 경우 아이디와 패스워드는 admin, admin 이며 로그인 완료 후 시스템관리-보고서 보안 사용자 추가에서 패스워드를 변경할 수 있습니다.  admin 계정은 삭제 불가능합니다.


로그인 다이얼로그에서 “자동 로그인 사용”을 체크하고 로그인하면 다음부터는 로그인 다이얼로그가 표시되지 않고 마지막으로 로그인한 아이디로 자동 로그인 하게 됩니다.

그림 (2) 자동 로그인 취소

6. 스케줄링

스케줄링 서버

                        

                                        스케줄링이란 RDEditor로 작성한 보고서를 RDServer 스케줄링서버에 등록 후

                                        특정시간에 데이터 파일을 생성하도록 하는 기능입니다.

<실행 전 점검 사항>

Ø  메일통지 기능을 사용하기 위해 RDServer IP가 스팸 IP로 등록되어 있지 않은지 확인

Ø  RDServer에 db.properties, rd.properties파일이 /RDServer/conf 아래에 있는지 확인하고 데이터베이스에 대한 설정이 맞는지 확인

Ø  JDK1.3 이상 (이하 버전에서는 스케줄링 기능을 사용할 수 없음)

 

<처리과정>

스케줄링 작업은 RDServer 관리자-스케줄링메뉴를 통해 등록하고 처리는 서버에 작동중인 스케줄링 서버가 수행, 처리 후 결과 파일(보고서와 데이터 파일)을 서버의 특정위치에 생성되도록 합니다. 옵션에 의해 메일통지 기능을 사용할 수 있습니다.

 

<스케줄링 제약사항>

1)스케줄링은 보고서 작성 시 DB접속방법이 RDAGENT인 경우만 가능

2)보고서 타입은 일반보고서, 표방식보고서에 대한 것만 정상지원

 (스크립트 및 저장프로시저 문서는 사용불가)

3)스케줄링을 사용하기 위한 지원 데이터베이스:
DB2, MSSQL, Oracle, MYSQL, Sybase

 

스케줄링 서버 시작

 

Ø  서블릿 버전: rdagent.jsp에서 schedule=true 설정 후 최초 접속 시 자동 Start (rdagent.jsp)

private static boolean schedule = true;

Ø  데몬 버전: 서버 구동 스크립트에서 플래그 셋팅 (Default =false)

startup.bat : set RDSERVER_SCHEDULER=true

startup.sh: RDSERVER_SCHEDULER=true

           스케줄링에 사용될 MRD 문서를 등록합니다.

 

n  올리기
  

왼쪽 트리창에서 서버의 지정 디렉토리를 선택하고, 올리기 버튼을 누릅니다.

 

 

 

파일선택: 찾아보기 버튼을 눌러 스케줄러 서버에 등록할 로컬 문서를 선택합니다. (200자까지 입력 가능)

 

디렉토리: 올려질 파일의 서버 디렉토리를 입력합니다.

디렉토리 지정을 하지 않을 경우는 서버의 루트디렉토리에 올려지며, 입력한 디렉토리가 서버에 존재하지 않을 경우에는 자동생성 됩니다.

(200자까지 입력 가능 )

 

파일설명:  파일설명은 입력하지 않아도 무방합니다.(20자까지 입력 가능)

 

 

 

n  삭제

서버에 등록된 MRD 파일을 삭제합니다.

 

n  디렉토리 생성, 삭제, 변경

디렉토리를 생성하여 디렉토리별로 파일을 올리고 관리할 수 있습니다.

 

 

n  디렉토리 권한설정

디렉토리 생성 후 각 디렉토리별로 접근권한을 설정할 수 있습니다.

기본적으로 디렉토리를 생성한 사용자와 admin 계정에게만 접근 권한이 있습니다.

디렉토리 권한설정은 해당 디렉토리에서 오른쪽 마우스 버튼을 누르면 나타나는 메뉴에서 설정할 수 있습니다.

Administrator 그룹의 사용자만이 디렉토리 권한설정을 할 수 있습니다.

         

 

n  전체 파일 보기

          서버에 업로드되어 있는 모든 파일을 보여줍니다.

 

 

 

 

 

스케줄링 새로 등록하기

스케줄링/새로 등록하기 메뉴를 선택합니다.

참고) 스케줄링 메뉴가 Disable 되어있는 경우는 서버가 Lite 버전 또는 스케줄링 지원 DB가 아니거나 서버구동 시 스케줄링 Start옵션 설정을 하지 않았을 경우입니다.

n  스케줄링 명

스케줄 등록 이름을 입력합니다. 최대 입력값은 40자입니다.

n  스케줄 대상 문서

스케줄링 할 문서를 찾아보기 버튼을 눌러 선택합니다. 목록에서 하나를 선택하면 문서경로가 자동으로 입력된 후 창이 닫힙니다. MRD 문서목록이 없을 경우는 파일등록 메뉴를 통해 문서 올리기 이후 수행합니다.

 

n  실행 파라미터

* 지원 파라미터: /rp, /rserverprn, /rpdrv

스케줄 문서에 필요한 /rp 파라미터 값들을 입력합니다.
    최대 입력값은 512자입니다.

파라미터에 공백을 구분자로 하여 직접 입력(데이터에 공백이 있을 경우 []로 묶음)하거나 파라미터를 가져올 jsp 또는 asp 패스 URL을 입력합니다.

u /rp 사용 예)

파라미터를 직접 입력할

경우

입력 예) /rp 20021218 20021230

사용자 정의 jsp 또는

asp 호출 결과를 파라미터

로 사용할 경우

 

입력 예) http://server:port/rpparam.jsp

 

jsp 내용 예)

out.print("/rp 20021218 20021230");

asp 내용 예)

Response.Write("/rp 20021218 20021230");

다중 파라미터를 입력할

경우

(구분자 ;)

입력 예) /rp 1 2;/rp 3 4

용도: 하나의 문서를 가지고 서로 다른

파라미터를 적용하여 결과파일을 생성하고자

할 때 사용하는 옵션입니다.

이와 관련하여 각각의 결과에 대해 메일수신자가

다른경우, 서로 대응되도록 세미콜론(;) 을

구분자로 하여 입력합니다.

 

u  /rserverprn,/rpdrv 사용 예)

서버프린트 체크 후 출력을 원하는 프린터를 선택하면 자동으로 파라미터 입력창에 입력됩니다. (서버프린트 기능은 서버O.S가 Windows만 가능)

 

 

n  스케줄링 결과 파일명: 스케줄링 후 생성될 결과 파일명을 입력합니다. 결과 파일들은 입력한 값을 prefix 로 하여 .txt, .mrd, .html 파일들을 생성합니다.

 

<참고사항>

결과 파일에 디렉토리를 지정하고 싶을 때는 디렉토리 패스함께 결과명을 입력하면 됩니다. 최대 입력 값은 40자입니다.

예)/myfolder/myDoc

myfolder라는 디렉토리 생성 후 myDoc생성날짜_일련번호 형식으로 파일을 생성합니다. 생성날짜는 년월일시분초랜덤번호로 구성되며(단, mrd 문서는 일련번호는 붙지 않음), 실제 서버의 디렉토리를 확인해 보면  RDServer/reports/data/사용자계정/myfolder 가 됩니다.

n  실행 파라미터 참조 index: 스케줄링 결과 데이터 파일명은 마지막에 일련번호가 붙습니다. 일련번호는 /rp 파라미터의 값으로 대체할 수 있습니다.  “실행 파라미터 참조 index”는 /rp 파라미터 중 몇 번째 파라미터로 대체할 것인지 지정해 주는 값입니다.

예를 들어 실행 파라미터를 /rp [1] [myid]로 주고, 실행 파라미터 참조 index를 2로 입력 하였다면 결과파일명은 OOOO_myid.html 와 같은 형태로 생성됩니다.

n  서버 프린트(Server Print)

서버 O.S가 windows일 경우, 사용 가능한 옵션으로 스케줄링 완료 후 선택한 프린터로 출력할 수 있습니다.

서버프린트 체크 후 출력할 프린터를 선택합니다.

 

n  스케줄 실행 기준일(스케줄 시작일):

스케줄링 시작 기준일을 시까지 입력합니다. (24시간 기준)

 

n  스케줄 실행 날짜:

정해진 날짜에 한번 실행하기: 정해진 날짜에 한번 실행합니다.

매주에 실행하기: 매주에 선택한 요일과 정해진 시간에 실행합니다.

매월에 실행하기: 매월 정해진 날짜에 한번 실행합니다.

매일 실행하기: 시작일로부터 매일 정해진 시간에 실행합니다.

매시간마다 실행하기: 정해진 시간, 매분에 실행합니다.

 

n  메일발송 여부:

스케줄링 완료 후 메일통지 여부를 선택 및 메일 수신자를 입력합니다. 받는 사람이 복수인 경우, 세미콜론(;)으로 열거합니다. 최대 입력값은 512자입니다.

만약 메일리스트를 jsp 또는 asp를 호출하여 가져올 때 다음의 예처럼 URL을 입력합니다.

예) http://ip:port/schedulemaillist.jsp

<참고사항> 실행파라미터에 다중파라미터 값을 입력한 경우, 세미콜론(;) 구분자로 수신자 입력란에 주소를 입력합니다. 그러면 /rp 입력 순서에 따라 차례로 매핑되어 메일이 발송됩니다. 실행파라미터에 대응되는 메일 입력값이 없을 경우, 메일은 발송되지 않습니다.

 

n  메일발송 형식: 메일발송 시 내용을 HTML 형식으로 embed시킬 때는 HTML를 선택하고 스케줄링 완료 파일 위치만 링크시킬 때는 URL LINK를 선택합니다.

- URL LINK 선택 시 스케줄링 메일통지 양식(HTML) 작성하기 참조

예) HTML 형식으로 메일을 보낸 경우 (기준: Outlook express 이며

내용에 RD ActiveX가 바로 보이는 형태)

 

<참고사항>

* HTML형식으로 보낼 때는 사용자가 사용한 메일 프로그램이 낮은 보안수준 옵션을 가져야 합니다.

예) Outlook expresss인 경우:

      도구/옵션/보안-인터넷영역(보안수준은 낮고 기능성은 높음)

* 스케줄링이 제대로 시행되지 않을 때에는 스케줄링/서비스제어 메뉴에서 스케줄링을 재시작 또는 일시중지시킬 수 있습니다.

* 스케줄링은 시작 되었으나 쿼리 및 데이터의 양에 따라 데이터 파일이 즉시 생성되지 않을 수도 있습니다. 서버 매니저의 로그/스케줄링 로그보기를 참조하여 스케줄링 관련 진행 상태를 파악할 수 있습니다.

스케줄링 메일발송

 

스케줄링 메일발송은 이미 스케줄링이 끝난 파일을 선택해서 메일을 보낼 수

있는 기능입니다.  $RDServerHome/ocxhtm/로그인계정 아래에 있는 html

파일들을 정의된 메일 아이디로 발송하게 됩니다.

 

 

 

n  소스파일: 발송해야 하는 정보가 담긴 파일 경로를 입력합니다. 입력제한

길이는128자입니다. 텍스트파일(서버의 절대경로) 또는 URL로 정의해 줄 수 있습니다.

 

소스파일 위치 입력 예)

Windows 계열: C:/m.txt

linux, unix 계열: /usr/local/RDServer/m.txt

 

소스파일의 내용은 발송해야 하는 대상 파일 위치와 메일수신자를 ,로 구분하여 정의합니다. 건당 구분자는 리턴 캐릭터로 정의해 줍니다.

 

/usr/local/RDServer/m.txt 소스 파일의 내용 예) 

admin/test2006030914000024_0.html,VINET@m2soft.com

admin/test2006030914000025_0.html,hssong@hotmail.com

n  발송제목: 발송시 제목을 입력합니다. 제목 최대길이는 30자 이내 입니다.

n  실행: 소스파일을 읽어, 메일 발송을 시작합니다. 서버에 기존에 진행 중인 작업이 있으면 실행하지 않습니다.

n  모니터링 시작: 발송되는 과정(성공, 실패)을 실시간 모니터링합니다. (실행이 처음으로 이루어질 때 기본값은 모니터링 시작)

n  모니터링 중지: 모니터링을 중지합니다.

7. 일괄작업 (Batch Job Processing)

일괄작업은 RDServer에 작업을 등록하고 실행하는 기능입니다.

작업에는 보고서 작업과 EJB작업이 있습니다.

보고서 작업은 서버에 등록되어 있는 보고서(MRD)를 실행시키고 실행결과를 Mail 또는 SMS로 전송시키는 기능입니다.

EJB 작업은 서블릿 서버에서 JNDI 명으로 EJB를 접근하여 호출/실행 할수 있는 기능입니다. EJB 작업등록을 하려면 J2EE 서버에 RDServer 서블릿 버전이 설치되어야 하고, 실행 가능한 EJB 리스트들을 JNDI 명으로 lookup 가능해야 합니다.

 

목록보기

서버에 등록된 작업을 보여줍니다.

 

 

n  Name: 작업명 (작업명은 중복될 수 없음)

n  Type: 작업의 종류를 보여줍니다.

n  Invoke: 작업을 실행합니다.

n  Update: 작업을 수정합니다.

n  Delete: 작업을 삭제합니다.

보고서 작업을 등록합니다.

보고서 작업은 스케줄링과 유사합니다.

단지 수행시간의 설정이 없습니다.

 

 

n  작업 이름: 작업명을 지정합니다. 작업이름은 중복될 수 없습니다.

n  작업 대상 문서: 서버에 등록되어있는 MRD 문서를 찾아보기 버튼을 눌러 지정합니다.

n  실행 파라미터: 보고서 파일의 실행 시 필요한 파라미터를 적어줍니다.

    사용가능 파라미터는 /rp, /rserverprn /rpdrv 입니다.

n  Server Print: OS가 Windows 계열인 경우 나타납니다. 작업완료 후 보고서를 지정된 프린터로 출력합니다.

n  메일 발송 여부: 보고서 실행후 결과를 메일로 발송할지 여부를 설정합니다.

n  수신자: 메일의 수신자 주소를 적어줍니다. 여러 개인 경우 ; 로 구분합니다.

n  발송형식: 메일의 발송형식을 지정합니다. HTML은 RD OCX Viewer를 메일본문에 직접 보여주는 방식이고, URL LINK는 보고서를 볼 수 있는 URL만 보내는 방식입니다. URL LINK 를 선택 시 HTML 템플릿을 환경설정/경로- 메일 발송 양식 HTML 에 지정할수 있습니다.

10. 시스템관리

캐쉬

캐싱 기능은 사용자로 하여금 데이터베이스에서 패치해온 데이터를 사용자가 지정한 시간 동안 서버의 메모리 또는 파일로 저장할 수 있게 해줍니다. 이와 같은 기능은 다음 번 사용자가 같은 쿼리로 요청할 때 서버에서 직접 데이터를 주는 것이 가능해지므로 데이터베이스의 부하와 리포팅 시간을 줄일 수가 있습니다. 캐싱되어 있는 데이터는 현재의 데이터가 아니라 지난 데이터이기 때문에 데이터에 따라 기간설정을 해주는 것이 중요합니다.

캐쉬 서버는 스케줄링 기능을 포함하고 있습니다. 사용자는 캐싱된 파일을 언제 갱신할 것인지 기간과 시간을 세팅할 수 있습니다. 이와 같은 기능은 첫 요청자가 리포팅을 수행할 때 서버쪽에서 캐싱도 함께해야 한다는 부담을 덜어줍니다.

캐쉬 서버를 구동 시키려면 쿼리 캐싱을 사용하는 보고서(DB접속방법: RDAGENT)가 한번 이상 수행되어야 합니다.

그림(2) 에디터-데이터베이스/쿼리정의-쿼리 캐싱 사용하기

n  시간: 앞에서 설정된 갱신 날짜가 될 때 캐싱을 실행할 시간입니다.

n  유효기간: 현재 설정된 유효기간 (갱신날짜) 입니다.

n  쿼리: 각 케이스를 대표하는 고유키로 사용됩니다.

기본 그룹은 Users와 Administrator로 나누어 지며 새로운 그룹을 추가하거나 삭제할 수 있습니다. 그룹명은 최대 30자입니다. 사용자를 추가할 때 반드시 그룹 중 하나를 선택해야 하며 그 그룹이 사용할 수 있는 서비스명을 추가시켜 주어야 합니다. 서비스명은 환경설정-JDBC 서비스에 정의된 이름들입니다.

서비스 추가

그림(2)는 추가 버튼을 눌렀을 때 나오는 설정 창을 보여줍니다. 여기에서 사용자는 추가될 새로운 서비스명을 입력하면 됩니다. 이미 추가되어 있는 동일한 서비스명을 입력했을 경우, “새로운 서비스” 창은 닫히지 않고 다시 표시됩니다.

새로운 서비스명을 적은 후에는 그림(3)과 같은 새로운 서비스 추가 위저드창이 표시됩니다. 사용자는 여기에서 새로운 서비스에 대한 설정 값을 입력할 수 있습니다.

새로운 서비스를 추가하였을 때 기본적으로 “해당 서비스로 아직 연결이 되지 않았습니다”라는 메시지가 표시됩니다.

그림(5)는 새로운 서비스를 추가하였을 때 적용 시기를 묻는 창입니다. 사용자의 선택 여부에 따라 즉시 적용되어 새로운 서비스에 대한 연결 시도를 하거나 서버 재구동시 연결시도를 하도록 설정할 수 있습니다.

즉시 적용을 선택하였다면 RDServer는 새로운 서비스의 등록정보를 이용하여 커낵션을 테스팅하고 실패했을 경우, 그 원인을 연결상태에서 보여줍니다.

재구동시 적용을 선택했을 경우, 해당 서비스의 탭은 빨간색으로 표시됩니다. 연결상태 창은 해당 서비스가 존재하지 않는다는 메시지를 보여줄 것입니다. (아직 업데이트가 되지 않았으므로 당연합니다.)

 

업데이트

확인을 누를 시에는 위의 그림(4)나 (5)와 같은 경우로 메시지 창이 표시될 수 있습니다. 변경된 사항이 있거나 해당 서비스의 연결상태가 실패일 경우에는 그림(5)처럼 “Connection Reloading” 창이 표시됩니다.

“Connection Reloading”은 “서버 재구동시 적용”과 “즉시 적용” 두 가지 옵션이 있습니다. 서버 재구동시 적용은 서버를 다시 리스타트 했을 때 DB와의 풀 설정을 변경하는 것이고 즉시 적용은 DB와의 모든 커낵션을 즉시 끊고 새로이 풀을 생성하여 접속하는 방법입니다.

확인버튼을 누름과 동시에 선택된 옵션이 실행됩니다.

사용자 정의 문자 셋: 서버의 언어 캐랙터 셋을 입력합니다. (EUC-KR, SHIFT_JIS, gb2312,8859_1), 정의 되어 있지 않으면 서버 내부의 기본 charset 을 따릅니다.

스케줄결과 파일 포멧(yyyyMMdd): 스케줄링된 결과파일명 포멧을 지정해 줍니다. TRUE일 경우는 yyyyMMdd, FALSE일 경우는 yyyyMMddhhmmss+serial 로 저장됩니다.

 

 

 

 

보고서

그림 (5) 보고서 설정화면

세션 타임아웃: 보고서에 대한 세션 타임아웃을 설정합니다. 단위는 분입니다. 보고서(RDViewer,RDViewer OCX)가 실행되는 도중 외부적으로 강제 종료되거나 비정상적으로 종료되어 응답이 없는 경우는 서버쪽 모니터링에 의해 데이터베이스 연결들을 자동 반납하게 합니다.

서버Export 최대 사용자 수: 서버 Export 사용 시 최대 동시 사용자 수를 지정합니다.

로그

설정 패널 왼쪽 상단에 보면 암호화 저장 체크박스가 있습니다. 로딩시 체크의 유무는 현재 파일의 암호화 상태를 나타내며 사용자가 체크상태를 바꾸어 주고 확인을 누름으로써 서버에 저장되어 있는 log.properties파일을 암호화 및 복호화시킬 수 있습니다.

각 로그에 대한 위치를 설정할 수 있으며 수정 후 적용은 서버 restart시에 가능합니다.

로그설정

로그파일의 경로, 로그파일 최대 개수, 로그 스위퍼 실행간격을 설정합니다. 로그 스위퍼란 로그파일의 최대 개수를 유지해주는 쓰레드를 말합니다. 지정된 실행 간격마다 쓰레드가 수행되면서 로그파일의 개수를 체크합니다.

로그파일 최대 파일수가 0이면 로그 스위퍼는 동작하지 않습니다. 즉 로그파일의 개수에는 제한이 없습니다.

 

에러로그

에러로그파일의 레벨과 출력방향을 지정합니다.

 

-          에러 로그 레벨: 에러를 기록하는 기본 level을 지정합니다.

(옵션 : debug > info > error > none )

-          에러 출력 방향: 에러를 File 또는 console로 보낼지를 설정합니다. 콘솔을 선택하면 오류가 표준 출력으로 디스플레이 됩니다.

 

보고서 사용자 로그

보고서 사용자 로그파일의 기록여부, 레벨, 서비스명을 지정합니다.

 

-          로그 기록여부:  기록여부 설정

-          로그 레벨: File(0) 또는 Database(1) 에 기록할지를 설정합니다.

-          서비스 이름(JDBC): 로그 기록을 Database(1)에 할 경우, 사용할 JDBC 서비스를 선택합니다. 서비스명은 환경설정-보고서-JDBC 서비스 참고.

리포트 상세 조회

 

-          로그매니저 사용여부: WAS 또는 데몬 서버 부팅 시 로그 매니저 구동여부를 정의합니다.

-          로그 방법 (메모리 기본 저장): 리포트 로그 정보의 저장 매체(파일 또는 DB)를 선택합니다. DB인 경우는 보고서 사용자 로그의 서비스 이름에 정의된 DB에 ReportLog 테이블이 생성되어 조회 정보가 저장됩니다.

-          리포트 로그 경로: 로그방법이 파일로 정의될 경우 정의된 파일경로에 리포트 로그 정보가 저장됩니다. 참고) 로그경로가 길어 정보창에 잘려서 보일 때는 입력창에 마우스를 클릭하고, 오른쪽으로 드래그하여 이동합니다.

-          Report Log Search Range: 로그 매니저 사용시 많은 양의 데이터가 로그 파일에 남습니다. 로그 매니저가 RDServer가 남긴 모든 로그 데이터를 분석할 경우 서버에 많은 부하를 줄 수 있습니다. 분석 하고자 하는 로그 데이터의 범위를 지정하여 서버의 부하를 줄일 수 있습니다. 검색 범위는 일(day) 단위로 지정합니다.

 

캐쉬

캐쉬 서버에 대한 설정입니다. 캐쉬 서버는 어떠한 보고서가 캐싱을 하겠다고 서버로 요청할 때 시작됩니다. 환경설정에서 변경된 캐쉬 서버관련 정보는 다음 번 WAS가 리부팅되고 캐쉬 서버가 시작될 때 적용됩니다.

첫 페이지 로딩 시 상단에 있는 암호화 저장 체크박스는 현재 cache.properties 파일의 암호화 여부를 나타냅니다. 확인버튼 적용 시 체크박스의 유무에 따라 서버 cache.properties의 암호화/복호화가 수행됩니다.

그림 (1) 캐쉬 환경설정 화면

설정 입력창에 포커스를 주었을 때는 그림(1) 처럼 옵션에 대한 도움말을 정보창에 표시하여 줍니다.

환경

n  캐싱 옵션: 기본적으로 사용할 옵션, 입력 값은 1~3 [1은 일 단위(시간), 2는 주 단위(요일과 시간), 3은 월 단위(일자, 시간)]

n  업데이트 일 간격/업데이트 날짜: defaultUpdateOption=1일 때 입력 값은 일 간격을 입력/defaultUpdateOption=3일 때 입력 값은 1~31(일자)

n  업데이트 요일: defaultUpdateOption=2일 때 유효함, 입력 값은 1(Sun) 2(Mon) 3(Tue) 4(Wed) 5(Thu) 6(Fri) 7(Sat)

n  업데이트 시간: defaultUpdateOption=1 or 2 or 3일 때 유효함, 입력 값은 1~24(시간)

n  쿼리 업데이트 간격 (분): 업데이트해야 할 캐쉬의 쿼리가 2개 이상일 때, 쿼리를 보내는 시간 간격을 설정합니다. 입력 값은 분 단위

파일

n  캐쉬 파일 저장 위치: 캐쉬 파일을 저장하는 서버의 물리적인 경로

n  각 캐쉬 폴더의 파일 수: 각 캐쉬 폴더당 저장하는 캐쉬 파일의 수.

스케줄링

n  업데이트 스케줄러 사용 여부: 캐쉬 파일의 업데이트 스케줄러를 사용할 것인지를 설정.

JVM

n  JVM 사용 여부: JVM의 메모리 사용 여부 (true or false).

n  JVM Total Memory대비 최대 사용률 (0~20%): JVM의 total memory중 20%의 cache memory를 사용, 입력 값은 0~20 시스템의 효율적인 사용을 위해 20% 이상 캐쉬 메모리로 할당하지 않음 (정수 값).

SAP

SAP R/3 의 RFC나 테이블를 이용하여 보고서를 작성하고 싶다면 문서 작성 시에 SAP GUI 클라이언트를 개발자 PC에 설치하여 문서를 작성해야 합니다.

RDServer는 런타임시에 SAP에 연결하여 데이터 패치 기능을 제공합니다. (서버 시작 시 sapjco 관련 클래스와 라이브러리가 클래스 패스에 추가되어야 함 – 자세한 설정 사항은 ServerManual 참조).

다음은 SAP 서버에 연결하기 위한 서버측 환경설정을 위해 제공하는 화면입니다.

상단에 보면 암호화 저장 체크박스가 있습니다. 로딩시 체크의 유무는 현재 파일의 암호화 상태를 나타내며 사용자가 체크상태를 변경하고 확인을 누름으로써 서버에 저장되어 있는 sap.properties파일을 암호화 및 복호화시킬 수 있습니다. 로그 파일 경로는 sap에 관련된 로그파일을 저장할 경로를 나타냅니다.

 

-          사용자: SAP Server에 등록된 사용자 아이디.

-          암호: SAP Server에 등록된 사용자의 암호.

-          클라이언트: 설치된 SAP Client의 아이디.

-          시스템 번호: 시스템 번호.
